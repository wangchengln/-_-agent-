Ch 2. 从零到一开发 Agent Skills 系统 二、借助 LangChain 开发 Agent Skills 系统入门

LangChain 1.x create_agent Core Tools Bootstrap 实战开发

本节目标：基于 LangChain 1.x 的最新标准，从零搭建一个支持 Agent Skills 的最小可行系统（MVP）。你将学会构建核心工具层、技能引导层和 Agent 运行时，理解 Skills 系统从"扫描技能"到"执行任务"的完整链路。

2.1 系统架构设计：MVP 版本

要实现一个真正支持 "Agent Skills" 的系统，我们不能只把目光停留在简单的对话链上。我们需要构建一个具有 "感知-决策-执行" 闭环的 Runtime（运行时）。

基于 LangChain 1.x 的最新标准（特别是 Graph-based 架构），我们将系统架构精简为以下三个核心模块。这个 MVP（最小可行性产品）版本虽然精简，但保留了 OpenClaw 最核心的"灵魂"：

🔍

1. 能力引导层 (Bootstrap)

在系统启动的瞬间，扫描本地 skills 文件夹，提取元数据，生成技能快照 (SKILLS_SNAPSHOT)。

🔧

2. 核心工具层 (Core Tools)

赋予 Agent "阅读文件" (Eyes) 和 "联网获取" (Hands) 的基础物理能力。

🧠

3. 智能体运行时 (Agent Runtime)

使用最新的 create_agent API 构建核心大脑，并动态注入 System Prompt，实现对技能的挂载。

架构流程图

Skills 文件夹 → 扫描器 (Scanner) → SKILLS_SNAPSHOT → System Prompt
用户指令 → Agent Runtime ⟶ read_file → SKILL.md ⟶ Agent Runtime ⟶ fetch_url → 互联网
Agent 先读取 SKILL.md 学习 SOP，再根据指令调用 fetch_url 执行任务

2.2 核心工具层 (Core Tools) 实现

Agent Skills 的核心逻辑是："Agent 通过阅读文档学会使用工具"。因此，在 Agent 出厂设置中，必须原生具备两个最基础的工具：read_file（用于读文档）和 fetch_url（用于执行文档里的联网请求）。我们使用 LangChain 的 @tool 装饰器进行封装：

import os
from langchain_core.tools import tool
from langchain_community.tools.file_management import ReadFileTool
from langchain_community.tools import RequestsGetTool

# --- 工具 1: 安全的文件读取 (The Eyes) ---
# 初始化底层工具，强制指定 root_dir 为项目根目录
# 安全警告：这是防止 Agent 越狱读取 /etc/passwd 等系统文件的关键！
raw_read_file = ReadFileTool(root_dir="./backend/skills")

@tool
def read_file(file_path: str) -> str:
    """
    用于读取 Agent Skills 的定义文件 (SKILL.md)。
    输入参数 file_path 必须是相对于 skills 文件夹的路径，
    例如: 'get_weather/SKILL.md'
    """
    try:
        # 在这里可以添加额外的路径检查逻辑
        if ".." in file_path:
            return "Error: 禁止访问父目录文件。"
        return raw_read_file.invoke({"file_path": file_path})
    except Exception as e:
        return f"Error reading file: {str(e)}"

# --- 工具 2: 联网能力 (The Hands) ---
# allow_dangerous_requests=True 是必须的，否则无法访问公网
raw_requests = RequestsGetTool(allow_dangerous_requests=True)

@tool
def fetch_url(url: str) -> str:
    """
    用于执行 HTTP GET 请求获取网页或 API 数据。
    当 Skill 文档要求访问某个 URL 时使用此工具。
    """
    try:
        # 实际生产建议：在此处集成 html2text 库，
        # 将 HTML 转为 Markdown，减少 Token 消耗
        response = raw_requests.invoke({"url": url})
        return response[:2000]  # 简单截断，防止网页内容过长
    except Exception as e:
        return f"Error fetching URL: {str(e)}"

# 将工具打包，准备注入 Agent
core_tools = [read_file, fetch_url]

代码深度解析

沙箱机制 (root_dir)：注意 ReadFileTool 的初始化。我们并没有让 Agent 满磁盘随便读，而是将它的视野通过 root_dir="./backend/skills" 锁死在技能目录中。这是 Agent 系统安全设计的第一道防线。
装饰器 (@tool)：LangChain 的 @tool 装饰器不仅是将函数转为工具，它还会自动读取函数的 Docstring（注释）。read_file 下面那段中文注释最终会被转换成 Prompt 告诉大模型，所以这段注释必须写得非常精准。
上下文保护 ([:2000])：在 fetch_url 中，我们做了简单的截断处理。真实网页往往包含大量无用信息，如果不做处理直接丢给 LLM，不仅浪费钱，还会挤占 Skill 指令的空间。

2.3 技能引导层 (Skill Bootstrap) 实现

Agent 怎么知道自己有哪些技能？我们需要在系统启动时，通过代码自动扫描目录。这被称为 "Bootstrap（引导过程）"。我们将编写一个 bootstrap.py，它的作用是生成一份"技能菜单"：

import os

def generate_skills_snapshot(skills_dir="./backend/skills"):
    """
    扫描 skills 文件夹，提取元数据，生成 XML 格式的快照
    """
    # 使用 XML 标签包裹，有助于 Claude/GPT 模型
    # 更精准地识别结构
    snapshot = "<available_skills>\n"

    # 1. 遍历 skills 目录下的所有子文件夹
    if not os.path.exists(skills_dir):
        return "<available_skills>No skills found.</available_skills>"

    for skill_name in os.listdir(skills_dir):
        skill_path = os.path.join(skills_dir, skill_name, "SKILL.md")

        # 2. 校验：只有包含 SKILL.md 的文件夹才被视为有效技能
        if os.path.exists(skill_path):
            # 3. 读取 SKILL.md 的内容
            # 实际项目中建议只读前几行元数据
            with open(skill_path, 'r', encoding='utf-8') as f:
                content = f.read()

            # 简单提取：实际生产中应使用 YAML Frontmatter 解析库
            snapshot += f"""  <skill>
    <name>{skill_name}</name>
    <path>{skill_name}/SKILL.md</path>
    <description>（该技能的动态描述）</description>
  </skill>\n"""

    snapshot += "</available_skills>"
    return snapshot

代码深度解析

为什么用 XML？注意 snapshot 变量使用了 <available_skills> 这种 XML 标签。在最新的 Prompt Engineering 实践中（特别是 Claude 系列），模型对 XML 结构的理解能力远强于 JSON 或纯文本列表。这能显著提高 Agent 检索技能的准确率。
动态扫描实现"热插拔"：这个函数的存在，实现了技能的热插拔。你只需要在文件夹里新建一个 get_stock 目录，下次运行代码时，这个函数就会自动把它加入到 Agent 的菜单里，完全不需要修改核心代码。
路径映射：注意 <path> 标签。我们告诉了 Agent 技能的名称，同时告诉了它文件在哪里。这为后续 Agent 调用 read_file 埋下了伏笔。

2.4 Agent 运行时构建 (LangChain 1.x Standard)

这是最关键的一步。在 LangChain 1.x 中，官方引入了基于 Graph 理念的 create_agent API，它自动封装了"思考-工具调用-再思考"的复杂循环。

from langchain.agents import create_agent  # LangChain 1.x 新标准 API
from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage

# 1. 初始化模型
# 建议使用 gpt-4o 或 claude-3-5-sonnet
# 因为它们对 Function Calling 的支持最稳健
model = ChatOpenAI(model="gpt-4o", temperature=0)

# 2. 动态构建 System Prompt (核心魔法)
# 将刚才生成的 Skill Snapshot 注入到提示词中
snapshot_content = generate_skills_snapshot()

# 这里定义了 Agent 的"元指令" (Meta-Instructions)
# 这段 Prompt 决定了 Agent 的行为模式
system_instruction = f"""你是一个基于 Agent Skills 架构的智能助手。

【能力清单】
你拥有以下技能列表 (SKILLS_SNAPSHOT)，
它们定义了你的专业能力：
{snapshot_content}

【核心协议 (CRITICAL)】
1. 当用户请求的任务匹配上述技能时，
   你**严禁**直接猜测操作步骤。
2. 你的第一步行动**必须**是调用 `read_file` 工具，
   读取该技能对应的 `<path>` 文件。
3. 读取文件后，严格按照文件中的 Steps (步骤)
   指导去执行任务。
"""

# 3. 构建 Agent Runtime
# create_agent 自动将 model 和 tools 绑定，
# 并处理工具调用的循环逻辑
agent_executor = create_agent(
    model=model,
    tools=core_tools,
    system_prompt=system_instruction,
)

# 4. 执行测试：模拟用户提问
if __name__ == "__main__":
    print(">>> Agent System Initialized.")
    print(">>> User: 帮我查一下北京现在的天气")

    # stream 方法允许我们看到 Agent 的思考过程
    events = agent_executor.stream(
        {"messages": [("user", "帮我查一下北京现在的天气")]},
        stream_mode="values"
    )

    for event in events:
        if "messages" in event:
            last_msg = event["messages"][-1]
            print(f"[{last_msg.type.upper()}]: {last_msg.content}")

代码深度解析

System Prompt 的动态注入：这是整个系统的灵魂。变量 snapshot_content 是运行时生成的，这意味着 Agent 的知识库是活的。每次新增或删除技能文件夹，Agent 的能力清单都会自动更新。
核心协议 (CRITICAL)：我们在 System Prompt 中用加粗语气写了一段"核心协议"。这是为了对抗大模型的幻觉。大模型有时候会"自作聪明"，觉得自己知道怎么查天气就直接编造 API。我们必须强制它："先读文件，再行动"。
create_agent API：这个 API 是 LangChain 1.x 的集大成者：
  • 自动识别 Model 是否支持 Tool Calling
  • 自动把 core_tools 里的函数签名转换为 OpenAI 格式的 JSON Schema
  • 自动管理对话历史（Memory），Tool 返回结果后自动喂回给模型进行下一轮推理
  • 开发者不再需要手写 while 循环来处理"多步执行"

2.5 FuFan-OpenClaw 需求文档

以上 MVP 版本展示了 Agent Skills 系统的核心原理。而 FuFan OpenClaw 项目则是在此基础上的企业级完整实现，增加了 7 个核心工具、SSE 流式通信、五层记忆架构、Canvas/A2UI 可视化输出等高级特性。

从 MVP 到 FuFan OpenClaw 的演进路径：

  • read_file + fetch_url 升级为沙箱版

  • 新增 python_repl（Python 执行）
  • 新增 search_knowledge_base（RAG 检索）
